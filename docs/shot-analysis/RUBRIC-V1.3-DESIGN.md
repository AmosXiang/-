| --- | --- | --- | --- |
| 1782541753229 | **5→8→8**(12→6→6) | **10→5→8**(0→7→4) | **5→8→8**(11→6→6) |
| 1783073819790 | 5→5→5(11→13→10) | 10→10→10(0→0→0) | 5→5→5(11→9→8) |

病理与 v1.1 的 identity 漂移完全同源:分子来自模型自由扫描,计数随机。

## 1. 预筛召回审计(零 API 成本,方法同 v1.2 第 0 节)

对库内 3 视频全部 22 份成功报告做多数标记(≥2 轮)召回审计,
词表两轮迭代后(原型:scripts/rubric-v13-prescreen-audit.mjs):

| 弱项组 | 视频1 | 视频2 | 视频3 | 候选占比范围 |
| --- | --- | --- | --- | --- |
| text_in_frame | 93.8% | 93.3% | 100% | 13.5%-32.8% |
| 运镜弱项(两项合并) | 100% | 100% | 100% | 20.0%-29.7% |
| graphics_overlay | 90.0% | 91.7% | 100% | 20.3%-29.5% |

**剩余"漏检"逐条核验后全部翻案**:仅 2 个独立镜头(视频1 shot25、视频2 shot59),
其历史标记的 evidence 原文均不属于该镜头——模型按 0-based 编号发射 shotRef,
"镜头25"的证据实为 shot26(手机屏幕银行到账,时间戳吻合),"镜头59"实为片尾
"未完待续"屏。两个真实所指镜头都在候选集内 → **三组有效召回 100%**。

**顺带发现的新漂移源**:模型自报镜头编号存在 off-by-one 错误(v1.0-v1.2 均有)。
v1.2 的 shotRef 格式硬化只能保证"镜头N"可解析,不能保证 N 正确;
只有服务端编号(候选集自带 shotIndex)能消除。这是推广本模式的独立论据。

## 2. 设计(v1.2 架构原样推广,少量扩展)

1. **知识库 v1.3.0**:三个维度增加 `scoredBy: "server"` + `prescreen` 词表字段
   (text/graphics 扫 description;运镜弱项扫 movement+description,并增加确定性规则
   "movement 含多段值(逗号/顿号分隔)= complex_camera_transition 候选")
   + 各自的 adjudicationFewShots(取真实库内镜头,含编号错位案例的正确归属版)。
2. **预筛**:prescreen.ts 泛化为按弱项组产出候选集(identity 的人物指称逻辑保留特化)。
3. **裁决调用**:单次调用扩展为对 (shotIndex, weaknessId) 对逐一表态,
   仍是封闭域 + 每对恰好一次 + 缺一判无效。运镜两个弱项在裁决中区分
   (long_tracking vs complex_transition),占比合并计算(与现行 metric 一致)。
4. **主报告调用**:四个维度的 scores/shotRisks 职责全部移除后,瘦身为仅产出
   improvements + summary(输入附服务端已算好的四维统计,保证叙述与数字一致);
   validator 相应硬化(发射任何 scores/shotRisks 判无效响应)。
5. **锚点区间差异**:stillFrame/postProduction 的区间是 30%/10%(identity/camera 是
   25%/10%),ratioToAnchorScore 参数化为按维度 anchors 驱动,不再硬编码。
6. **cameraMotion 的 calibrationStatus: unverified 保留**:占比与锚点分虽确定性化,
   报告文字仍禁止出现成功率数字(该约束移入裁决 prompt 与服务端 evidence 模板)。

## 3. 漂移预期与验证方法

- 被消除:四维度全部计数/分母/占比/映射/算术漂移 + shotRef 编号错位。
- 残余:逐候选裁决翻转(v1.2 实测 3.3%-20%,存在轮次相关性)——同样只披露不平滑,
  每维度输出 hitsToNextBand。
- 验证协议:3 视频 × 3 轮;通过线 = **四个维度分数全部零变化**(overallScore 因此
  也应零变化,这是 v1.3 的标志性成果);成本预算 ≈ $0.5,usageMetadata 实测核对。

## 4. 成本影响

裁决调用输入增大(候选对 40-90 个 + 三组口径与示例,估 +2-4k tokens);
主调用大幅瘦身(移除全部逐镜头扫描与评分职责,输出从 ~5-17k 降到 ~1-3k)。
净变化预计:**持平至下降**(v1.2 实测单集 $0.048,预计 v1.3 ≤ $0.05)。

## 6. 待确认项

1. **主调用瘦身幅度**:方案 4 保留主调用只产 improvements/summary。更激进的选项是
   合并进裁决调用(单调用出全报告),省一次调用但职责混杂;保守选项是维持现状不动
   (只推广三个维度的算分)。推荐按方案 4。
2. **词表过拟合风险**:词表在 3 条库内视频上迭代到有效召回 100%,新视频的生僻表述
   可能漏筛(候选集封闭 → 漏筛不可挽回)。缓解:词表为知识库字段可持续补充;
   是否接受与 v1.2 测量边界同类的这一风险?
3. graphics 词表含"效果/光芒/激光"等泛化词,候选占比可能在特效密集的片子上升到
   40%+(成本小幅上升,裁决负担加大),是否接受宁多勿漏的取向?

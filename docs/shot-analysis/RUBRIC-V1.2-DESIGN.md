## 0. 设计已用真实数据预验证(零 API 成本)

在写本方案前,先把预筛规则做成原型,对库内 3 条视频的 **12 份历史报告**
(shot_analysis_reports 表)做了召回审计:历史上被模型标记过的 multi_character_interaction
镜头,预筛候选集能否覆盖。以"多数标记"(≥2 轮标记,过滤单轮噪声)为基准:

| 预筛规则版本 | 视频1 | 视频2 | 视频3 |
| --- | --- | --- | --- |
| 朴素版:仅注册角色名共现 | 71.4% | 64.3% | 60.0% |
| 扩展版(见下) | **100%** | **78.6%** | **100%** |

朴素版被审计否决:分镜描述大量用**角色功能词**指人(中介经理/房东/胖子/黑人队长/
一名士兵),不在 characters[] 注册表中。扩展版补上了这一层。

视频 2 剩余 3 个"漏检"逐条核验后定性:2 个是**模型过标**(张杰单人讲解镜头,
按判定口径"正反打单人画面不计"本就不该算,模型 4 轮里只有 2 轮标它——这正是要消除的
漂移本身);1 个(shot35"老太太上前恳求")是**文本未提及第二人但画面可能有人**的
真边界案例——见第 2 节"测量边界"。

## 1. 服务端预筛规则(确定性,纯字符串/正则,零模型调用)

**人物指称提取**,三类信号取并集,一个镜头的 description 中独立指称跨度 ≥2 即为候选:

1. **注册别名**:characters[].name 归一化——按 `/` 拆分("路人甲 / 胖子"→两个别名),
   按括号拆分("梅 (Mei)"→"梅"+"Mei"),去除长度 <2 的碎片;
2. **角色功能词词表**:队长|士兵|经理|中介|房东|老板|老太太|老头|胖子|男子|女子|
   少年|少女|路人|保安|医生|警察|守卫|教授|新人|幸存者…(固定词表,入库为知识库字段,
   可维护);
3. **复数/互动线索**:两人|三人|众人|人群|队伍|彼此|互相|对视|相拥|扭打|对峙|
   他们|她们|双方|(数)名/位 等(同为知识库字段)。

**跨度合并**:同一指称的组成词相邻/重叠时合并为一个人("黑人男性士兵"=1 人)。
仅在紧邻(gap=0)时合并——审计发现 gap≤2 会把"郑吒转向詹岚"误并成 1 人,已修正。

候选占比实测:全片镜头的 33%-51%,裁决调用的输入规模可控。

**会不会漏(如实回答)**:会,且是原理性的——描述文本只写了一个人、
但画面实际有多人的镜头(如 shot35),**任何纯文本方法(包括现在的模型自由扫描)都
无法可靠判定**;现状是模型靠对话上下文"猜",猜得不稳定(此类镜头 4 轮只有 1-2 轮被标),
这正是漂移源。v1.2 的选择:把 metric 显式操作化为
**"分镜描述文本中可辨识的多人同框"**,承认这是 analysis_json 输入模式的测量边界,
写入知识库 metric 字段。治本方案(可选配套项):增强上游 /api/analyze 的分析 prompt,
要求 description 明确列出画面内出现的所有人物——让文本承载事实,预筛自然受益。

## 2. 模型强制逐候选裁决(prompt/schema 设计)

独立的小裁决调用(与主报告调用分离):

- **输入**:仅候选镜头的编号列表 `[{shotIndex, timestamp, description}]`
  + 判定口径原文 + 2-3 个来自真实库内镜头的 few-shot 边界示例
  (一个命中、一个"正反打单人不算"、一个"背景路人不算");
- **输出 schema**:`verdicts: [{shotIndex: INTEGER, verdict: 'hit'|'no_hit', reason: STRING}]`,
  服务端校验**每个候选 index 恰好出现一次**——缺一个即 GEMINI_INVALID_RESPONSE。
  强制逐个表态,杜绝自由扫描模式下的"懒扫描"(视频3 曾出现 12→3 的计数崩塌);
- **禁止提名候选外镜头**:候选集是封闭域。理由:审计表明候选外的模型标记
  几乎全是过标或文本无据的猜测;
- **计数与评分全部服务端**:占比 = |hit| / 总镜头数,锚点映射、维度分、加权总分
  均由服务端计算,模型不再为该维度输出任何分数。报告的该维度 evidence 由服务端
  生成(附命中清单与占比),shotRisks 中的 multi_character_interaction 条目也由
  服务端按 hit 清单 + 知识库 recommendation + 模型 reason 合成。

主报告调用同步瘦身:其 prompt 与 schema 中移除 multi_character_interaction 的
扫描职责(其余弱项不变,作为 v1.3 是否推广此模式的对照组)。

## 3. 漂移预期与验证方法

**被确定性消除的漂移源**(这些在 v1.1 中全部实测发生过):
候选集漂移、分母漂移、懒扫描计数崩塌(12→3)、候选外误标(过标)、
锚点映射与算术误差、pulid_latency 噪声(见第 4 节)。

**残余漂移源(唯一)**:单个候选镜头的语义裁决翻转(hit↔no_hit)。
不承诺具体数字——但给出可验证的判定线:

- **验证协议**:3 条库内视频 × 3 轮独立运行(约 9 次裁决调用 + 9 次主调用,≈$0.5);
- **通过线**:每视频 3 轮 identity 分数零变化;同时完整披露逐候选翻转率
  (每个候选在 3 轮中的裁决一致性)与各视频占比到区间边界的距离;
- **已知风险预先声明**:视频 2 的历史多数占比 ≈23%,天然贴近 25% 区间边界
  (再多 2 个 hit 就跨带)。若它因 1-2 个候选翻转而跨带,那是**边界邻近**问题而非
  机制失效——报告会输出"距边界镜头数",消费方可见分数的置信边际。
  不做迟滞/平滑(保确定性),只做披露。

## 4. pulid_latency 噪声修复

**根因**:它本来就不是逐镜头弱项,而是聚合属性(身份锁定镜头占比),
让模型逐镜头发射它,发射量随机(实测 43→0、61→24)。

**修复**:
1. 知识库中 pulid_latency 增加 `reportLevel: "aggregate"` 标记;
2. 模型侧职责移除:裁决/主报告 prompt 均不再要求(也不允许)输出 pulid_latency 的
   shotRisks;validator 遇到模型发射的 pulid_latency 条目判 GEMINI_INVALID_RESPONSE;
3. 服务端确定性计算:身份锁定镜头 = description 含 ≥1 个注册角色别名(纯字符串判定),
   占比 >60% 时在报告顶层输出一条 aggregate advisory(含占比 + 知识库 recommendation),
   不进 shotRisks。阈值 60% 为设计判断(evidenceBasis: design),写入知识库可调。

## 5. 顺带硬化(同批实施)

- **shotRef 格式漂移**:历史 12 轮中有 1 轮输出纯时间戳导致不可解析。identity 维度
  改用服务端编号后此问题对该维度消失;其余弱项的 shotRef 在 validator 中强制
  `镜头\d+` 前缀,不合格判无效响应;
- 知识库版本升至 **1.2.0**;新增字段(人物指称词表、复数线索词表、reportLevel、
  操作化 metric 措辞)全部落在 replicability-rubric.json,无 SQLite schema 变更,
  无迁移需求;narrative 路径零改动。

## 6. 成本影响

裁决调用输入仅候选镜头(20-38 个)+ 口径 + 示例 ≈ 3-6k tokens,输出 1-2k;
主调用因移除多人扫描职责而略降。净变化预计 ≈ 持平至 +$0.01/集(默认 thinking 配置,
遵循 thinkingBudget 实验结论不动思考预算)。验证轮全程记录 usageMetadata 实测核对。

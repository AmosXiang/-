# 评估:是否为 shot-analysis 增加"制作复刻度"分析维度

状态:**已确认(2026-07-06)。** 用户认可"窄化为本链路可生产性"定位,全粒度制作考古不做。
弱项清单已基于 comfyui_tasks 真实数据挖掘并经用户逐条确认(手部=非弱项;文字入镜=实测痛点;
Veo 运镜=公开文档假设、未校准;PuLID 超时=数据采纳),证据档案见 COMFYUI-WEAKNESS-FINDINGS.md。
实现按 commit A(知识库/schema)→ B(analysisType + 迁移 002)→ C(真实验收+成本核对)推进。

溯源声明:本评估的外部参考是一份公开方法论描述(小红书博主关于"用 Claude Code 拉片"
的截图总结转述),**不是可获取的代码项目,没有阅读过任何源码**。下文所有设计均为
自行设计;该方法论中的成本数字(30 分钟视频 / Opus 4.8 / 约 $25)仅作为
"不要做成什么样子"的反面参考,不用于任何估算。

## 结论(先行)

**部分值得做。** 建议:

- **不做**:小红书式全粒度制作考古——逐 15 秒时间码级切点清单、台词/图形/作用
  三栏说明、图形层动效清单、工具链推断。理由见 Q1:与我们已有数据高度重叠、
  服务对象错位(供人类剪辑师复刻人拍视频)、且正是那个 $25 成本反模式的来源。
- **值得做(窄化变体)**:"**本链路可生产性**"评分——不问"别人怎么做的",
  只问"这套分镜我们的 ComfyUI + Veo 链路能不能做出来、哪里会翻车"。
  输入复用已有的 analysis_json(零新增视频调用),复用证据先行+锚定量表+封闭 id 域
  的既有设计模式,每集新增成本约 **$0.02-0.05**(估算依据见 Q2)。

## Q1:增量价值评估

先看重叠矩阵——"标准拉片六步"的分析对象与我们已有能力的对照:

| 方法论维度 | 我们已有的对应物 | 重叠度 |
| --- | --- | --- |
| 镜头语言/运镜 | `/api/analyze` 每镜头已产出 `movement` 字段 | 高(已有) |
| 构图/机位/景别 | 同上,`composition` 字段 | 高(已有) |
| 节奏/切点 | shot-analysis 的 `shot_rhythm` + `reversal_cadence` 模式 | 高(已有) |
| 定性判断/评分 | shot-analysis 六维锚定量表 | 高(已有) |
| 图形层动效清单 | 无 | 无重叠,但见下 |
| 工具链推断 | 无 | 无重叠,但见下 |

"图形层动效清单"和"工具链推断"确实是空白,但它们回答的问题是
**"人类团队用什么工具复刻这条人拍视频"**——而我们的生产链路是
剧本(Gemini)→ 分镜图(ComfyUI)→ 视频片段(Veo)。推断出"这条视频用了 AE 粒子
特效+剪映花字"对这条链路没有可执行的输入。为完整性硬加,就是在为不存在的用户做功能。

对**短剧生产链路的自我评估/迭代**这一实际需求,真正的空白是:
分镜脚本生成后、进 ComfyUI/Veo 之前,没有任何环节回答
**"这些镜头我们的链路做不做得出来"**。comfyui_tasks 表里真实存在的重试/失败记录
说明这个痛点存在。窄化变体的四个维度草案(实现阶段再细化):

1. **静帧可生成性**:每个分镜的画面描述是否在 ComfyUI 预设(01-04 工作流)能力圈内
   (多人交互、复杂手部动作、文字道具是已知弱项);
2. **运镜可行性**:`movement` 字段要求的运镜能否被 Veo 8 秒片段可靠实现
   (长跟踪镜头、复杂机位变化是已知弱项,应建议拆分);
3. **角色一致性压力**:跨镜头同角色出现密度,评估身份锁定(PuLID/参考图)链路的压力点;
4. **后期需求**:哪些镜头必须依赖生成之外的后期(字幕卡、图形层),明确标注而非假装能生成。

产出直接可执行:"镜头 12 需要 8 秒以上跟踪运镜,建议拆成两个静态镜头"这类建议
能直接回灌分镜脚本迭代。这与已有的钩子/剧情结构维度(回答"内容好不好")正交,
无重叠。

## Q2:实现方式与成本量级

**完全复用既有设计模式,只换分析对象**:新增一个知识库文件
(`replicability-rubric.json`,含四维度的锚定量表 + 本链路已知弱项清单作为"信号词"),
schema 沿用证据先行(每个判定必须引用镜头号/时间戳)+ 封闭 id 域 + 服务端加权重算。
作为可选的 `analysisType` 参数(默认仍是现有的叙事分析),独立调用,不合并进现有
prompt——合并会拉长 schema、增加现有报告的失败率,为省 $0.01 不值得。

**成本估算**(依据:我们的真实运行数据 + Gemini 2.5 Flash 现价
$0.30/M 输入、$2.50/M 输出;中文按 1 字符≈1 token 取保守上界):

| 项 | 量 | 成本 |
| --- | --- | --- |
| 输入(知识库约 1 万字符 + 74 镜头分镜 JSON 约 1.4 万字符 + 规则) | ≈3 万 token | $0.009 |
| 输出(参照上次真实运行 4.9 千字符) | ≈6 千 token | $0.015 |
| **每集每次(analysis_json 模式)** | | **≈$0.02,上浮到 $0.05 仍成立** |
| 全库 3 条视频跑一轮 | | <$0.15 |

对比反面参考的 $25/30 分钟:**低 3 个数量级**。关键原因不是模型便宜,而是
**不重复消费视频模态**——分镜 JSON 是 `/api/analyze` 已经付过一次视频分析成本的产物,
新维度纯文本进出。

## Q3:"每 15 秒语义切点"预切分作为降本手段——技术判断

**可行,但在我们的素材长度下是负收益,不建议采用。**

技术可行性本身没问题:ffmpeg 本机可用(ffprobe 已实测在 PATH 上),
`select='gt(scene,0.4)'` 场景切换检测对短剧以硬切为主的剪辑风格是可靠的;
Gemini API 也支持 `videoMetadata` 的 `start_offset/end_offset` 定向窗口分析。
管道在工程上成立。

但算一笔账(以 405 秒的 test a 为例,视频 258 token/秒):

| 方案 | 输入 token | 成本 |
| --- | --- | --- |
| 整段单次调用 | 405s×258 ≈ 10.4 万 + 知识库 1.5 万 ≈ 12 万 | ≈$0.036 |
| 按 15 秒切 27 窗、逐窗调用 | 27×(知识库 1.5 万 + 窗口 0.4 万) ≈ 51 万 | ≈$0.15 |

逐窗调用**贵 4 倍**:知识库 prompt 在每个窗口重复计费,窗口视频本身反而是小头。
只分析少数关键窗口可以省一点,但会丢失全局结构上下文(节奏/情绪曲线本来就是
全局属性),得不偿失。该方法论的 15 秒切点思路服务的是 30 分钟以上长视频
(那时单次调用撞上下文上限,切分是必需品)——那不是我们的场景。

**真正的降本手段我们已经在用**:优先走 analysis_json 模式,视频模态的钱在
`/api/analyze` 只付一次,后续任何维度的分析都是纯文本。ffmpeg 场景检测值得留作
另一个用途(校验 Gemini 切点对齐、为 ComfyUI 抽关键帧),但那是独立需求,不是本任务。

## Q4:现有封装限制是否成为瓶颈

**不会。** 实测素材库(本机 ffprobe/文件系统数据):

| 素材 | 时长 | 大小 |
| --- | --- | --- |
| 37900191798 | 154s(2.6 分钟) | 20.5MB |
| 36726377162 | 414s(6.9 分钟) | 45MB |
| test a | 405s(6.8 分钟) | 46MB |

典型素材 2.5-7 分钟 / 20-46MB,距 500MB 上限有 10 倍以上余量
(500MB 按同码率折算约 45-75 分钟)。且推荐路径(analysis_json)根本不上传视频。
无分片能力这一限制只在"整集 30 分钟以上长视频直接喂给模型"的场景下才会触发——
该场景已在 Q3 中判定为不采用。

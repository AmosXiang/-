# 上游 /api/analyze 增强设计:逐镜头"画面事实"结构化字段

状态:**设计文档(prompt 与 schema 文本已定稿),未实现,验证需配额。**
立项来源:rubric v1.2 拍板时定为"单独排期"的治本项;v1.2/v1.3 复验中实测的两个测量
边界案例是直接动机。影响面:所有**新**入库视频;库内旧视频通过下游回退保持兼容。

## 0. 动机:两个实测测量边界,病根都在上游

拉片可生产性评分(rubric v1.2+)以分镜描述文本为唯一证据源,由此产生两类
"文本没说、画面存在"的原理性盲区,均有实测案例:

1. **在场人物不全**(identity 维度):视频2 shot35「老太太赶紧上前,惊恐地恳求
   "别打了,别打了!"」——画面显然还有被打者与打人者,文本只写了一人。
   模型自由扫描时代靠上下文"猜"(4 轮只有 1-2 轮标它),v1.2 显式排除出 metric。
2. **物件与内容不分**(text/graphics 维度):v1.3.0 复验中 graphics 裁决命中
   11→5→6→12 的翻转,全部集中在名片/图纸/笔记本电脑类**物件镜头**——描述文本
   没有说"上面有什么、需不需要读出来",裁决只能猜。v1.3.1 用"物件≠内容"口径
   钉住了裁决端,但事实本身仍缺失:一张真的需要读出内容的名片会被误判 no_hit。

下游(裁决口径)能做的都做完了;剩余误差只能由上游把事实写进数据来消除。

## 1. 变更设计

### 1.1 schema:shots[] 新增三个结构化字段(定稿)

```js
presentCharacters: {
  type: 'ARRAY', items: { type: 'STRING' },
  description: '该镜头画面内实际出现的所有人物指称,包括无名人物(用功能词指称,如"中介经理""老太太""一名士兵");仅台词提及但画面未出现的人物不得列入;确实无人物时为空数组'
},
onScreenText: {
  type: 'STRING',
  description: '画面中需要观众清晰读出的文字内容(招牌、字幕卡、屏幕显示、信件/名片/图纸上的可读文字),如实转写;画面有文字物件但内容无需辨认、或没有文字时,输出空字符串'
},
overlayGraphics: {
  type: 'STRING',
  description: '画面中必须由图形层制作呈现的元素及其内容(UI 界面、弹窗、地图/定位界面、字幕卡、片头动画、特效字);没有则输出空字符串'
}
```

三字段加入 shots items 的 `required`(responseSchema 强制,与现有字段同级;
Gemini 结构化输出保证产出,空值语义明确,不会推高失败率)。
运镜事实已有 `movement` 字段承载,不新增。

### 1.2 prompt:新增第 4 节(定稿,追加在现有第 3 节之后)

```
4. 画面事实(逐镜头如实记录,用于下游可生产性评估):
   - presentCharacters:列出该镜头画面内实际出现的所有人物,包括无名人物(用功能词
     指称,如"中介经理""老太太""一名士兵")。仅台词提及但画面中未出现的人物不得列入。
   - onScreenText:画面中需要观众清晰读出的文字,如实转写其内容;文字物件入镜但内容
     无需辨认(如黑屏电脑、合上的信封)输出空字符串。
   - overlayGraphics:必须由图形层制作的画面元素(UI/弹窗/地图界面/字幕卡/片头动画/
     特效字)及其内容;没有输出空字符串。
   - description 的情节描述必须与 presentCharacters 一致:描述中提到的在场人物
     不得少于该列表;不得把台词提及误写为在场。
```

shortDramaMode 附加句与 storyboard enrichment 强制句保持不变,新段落位于两者之前。

## 2. 下游消费(rubric v1.4 候选,本设计不实施)

新字段存在时预筛/裁决优先消费,缺失时**回退现行词表路径**(旧视频零迁移):

| 维度/弱项 | 现行(v1.3.1) | 新字段可达 |
| --- | --- | --- |
| multi_character_interaction | 词表指称提取→候选→裁决"同框+互动" | `presentCharacters.length ≥ 2` 直接给出**同框事实**(确定性),裁决只剩"是否互动"一问,预期翻转面大幅收窄 |
| text_in_frame | 词表候选→裁决"是否需可读" | `onScreenText !== ''` 即命中——**弱项可能完全确定性化**(零裁决) |
| graphics_overlay_dependency | 词表候选→裁决"物件≠内容" | `overlayGraphics !== ''` 即命中——同上 |
| pulid_latency 聚合 | description 含注册别名 | `presentCharacters` 与注册表求交,身份锁定统计更准 |

测量边界的升级:从「分镜描述**文本可辨识**」→「分析模型对**画面在场事实**的记录」。
仍非绝对真值(analyze 模型也会看漏),但事实来源从"叙事文本的副产品"变成
"被 schema 强制的逐镜头交代",且一次记录、全部下游受益(分镜生成的 prompt
同样拿到更完整的画面事实)。

## 3. 兼容性

- **旧视频**:库内 3 条视频无新字段 → 下游按字段缺失回退词表路径,报告口径
  由 kbVersion 区分;不强制重分析(重分析会生成新 videoId 记录,旧报告关联不受影响)。
- **narrative 拉片、分镜生成、ComfyUI 链路**:只读取既有字段,新增字段无破坏;
  description 措辞增强只会让生成 prompt 的画面事实更完整。
- **assertStoryboardEnrichment**:不涉及(enrichment 字段族独立)。

## 4. 成本

analyze 每视频 1 次调用;新增字段预计 +5-15% 输出 token(60-74 镜头 × 三个短字段),
输入端 prompt +约 200 token。绝对值:free tier 内可忽略;付费 flash 约 +$0.002/视频。

## 5. 验证协议(需配额,约 4-6 次请求)

1. 重分析库内 1-2 条视频(各 1 请求),不入库替换,仅对照:
   - shot35 类已知边界镜头的 `presentCharacters` 是否补全第二人;
   - 名片/图纸/笔记本镜头的 `onScreenText`/`overlayGraphics` 是否正确区分物件与内容;
   - description 与 presentCharacters 的一致性抽查。
2. 用新产出跑一次 replicability(2 请求),对照 v1.3.1 报告:候选集收窄幅度、
   hit 集合变化逐镜头核对。
3. 数据合格 → 立项 rubric v1.4(下游消费 + 弱项确定性化);不合格 → 只保留
   字段作为参考信息,下游不消费。
